## 概要
前作では単純なCRUD処理に留まっていましたが、本アプリではタスクに状態（完了 / 未完了）を持たせ、それに応じて表示を切り替える仕組みを実装しました。
状態管理とUIの分離を意識することで、より実用的で拡張性の高いアプリケーション設計に挑戦しています。

----

## 設計思想
本アプリでは「シンプルで迷わない操作」を重視しました。  
機能を最小限に絞ることで、ユーザーが直感的に使える設計にしています。
また、状態（データ）とUI（表示）を分離することで、状態変更後に必ずrender関数で再描画する構造にしています。
これにより、表示とデータの整合性を常に保てるようにしました。

----

## 作成した背景
前作ではシンプルなCRUD処理を実装しましたが、
「状態管理」や「条件による表示制御」などの理解が不十分だと感じました。
そこで本アプリでは、
- タスクの状態（未完了 / 完了）
- フィルタリング機能
を実装し、より実用的なアプリ開発に挑戦しました。

----

## 機能
* タスクの追加ができます。
* タスクの削除ができます。
* タスクの完了 / 未完了の切り替えができます。
* フィルター機能（すべて / 未完了 / 完了）があります。
* localStorageによるデータ永続化に対応しています（リロード後も保持されます）。
* 期限（締切日）の設定をすることができます。

----

## 使い方
1. 入力欄にタスクを入力します。
2. 「追加」ボタンをクリックします。
3. タスクをクリックすると完了 / 未完了が切り替えられます。
4. 「削除」ボタンでタスクを削除できます。
5. フィルターで表示を切り替えることができます。

## 工夫した点
* フィルター機能を実装し、状態に応じた表示切り替えができるようにし、ユーザーがタスクを管理しやすいようにしました。
* completedフラグでタスクの状態を管理できるようにしています。
* タスク追加後に現在のフィルター状態によっては表示されない問題がありました。そのため、追加時にフィルターを「すべて」に戻すことで、
　ユーザーが混乱しないようにしました。
* CSSで全て、完了、未完了のうち選択したものに色を付けることでアクティブ状態を視覚的に分かりやすく表現しました。
* タスクの期限をもたせることで、実用性を向上させました。

----

## 設計・実装のポイント
- 配列とオブジェクトを使い、タスクの状態（完了 / 未完了）を管理できるようにしました。
- 状態変更のたびに部分更新すると不整合が起きやすいため、render関数でUIを一括再描画する設計にしました。(前作から継続)
- save関数とrender関数にわけることで、状態の更新とUIの描画を分離することができ、コードの可読性と拡張性を高めています。

----

## 苦労した点
フィルター機能において、「表示の状態」と「タスクの状態管理」を両立させることに苦労しました。 
当初は条件分岐が複雑になり、表示とデータの整合性が崩れる問題がありました。
そこで、表示に関する処理をrender関数に集約し、状態の変更はイベント処理側で行うように役割を分離しました。
その結果、コードの見通しが良くなり、バグも減らすことができました。
