## 概要
前作では単純なCRUD処理に留まっていましたが、本アプリではタスクに状態（完了 / 未完了）を持たせ、それに応じて表示を切り替える仕組みを実装しました。
また、アプリ開発を進めていく中で

- UIとロジックが混在すると可読性が低下
- 機能追加時に影響範囲が広がり、修正が複雑

といったことがあり、実際に機能追加のたびにコードの見通しが悪くなっていくのを感じました。状態管理とUIを分離し、より実用的で拡張性の高いアプリケーション設計に挑戦しています。

----

## 技術選定
* JavaScript（Vanilla JS）を採用しました

フレームワークを使うと状態管理が簡単になりますが、
仕組みの理解が浅くなると感じたため、
あえて素のJavaScriptで実装しました。

その結果、
- データの流れ
- UI更新の仕組み

を自分で設計・実装する経験を得ることができました。

## 設計

本アプリでは「データ管理」と「UI操作の分離」を重視して設計しました。

### 背景
初期実装では、render関数の中にデータ操作とUI処理が混在しており、
機能追加を行うたびにコードが複雑になっていきました。

実際に、
- どこでデータが変更されているのか分かりにくい
- 条件分岐が増え、表示とデータの整合性が崩れる
- 修正時に別の部分へ影響が出る

といった問題が発生しました。

### 改善
そこで、TaskManagerクラスとStorageクラスを導入し、
データ操作や永続化のメソッドを呼び出す形にしました。

- TaskManager：データの追加・削除・編集・タスクの状態の管理
- Storage ：保存、データの読み込み
- render：UIの描画のみ担当

また、処理の流れを以下のように統一しました。

1. ユーザー操作
2. データ更新（TaskManager）
3. localStorageに保存
4. renderで再描画

この一方向のデータフローにすることで、状態とUIの整合性を保つ設計にしています。

### 苦労した点
TaskManageやStorageを導入する際、
「どこまでをどのクラスに持たせるべきか」の判断に苦労しました。

特に、save関数との役割の違いが曖昧で、
最初はデータ保存の処理をどこに書くべきか分からず、
設計がブレてしまいました。

また、
データ更新とUI更新の責務の切り分けも理解しきれておらず、
renderをどのタイミングで呼ぶべきか混乱しました。

そのため、
既存コードと比較しながら整理し、
「データを変更する処理」と「表示する処理」を一つずつ分離していきました。

また、render関数の肥大化にも課題がありました。

機能追加を進める中で、
render内の処理が増え続け、
「どこを修正すればいいのか分かりにくい」状態になりました。

そのため、以下の単位で分割しました。

- render
- createTaskItem
- createNormalUI
- createEditUI
- createButtonGroup

分離しすぎると逆に可読性が下がるため、
「1つの関数に1つの役割」を意識しながら、
実際にコードを書き換えつつ適切な粒度を探しました。

### 結果
- データの流れが明確になった
- UI変更がデータに影響しなくなった
- 機能追加がしやすくなった

また、設計を意識することで、
単に動くコードではなく「保守しやすいコード」を書く重要性を理解しました。

## 機能
* タスクの追加ができます。
* タスクの削除ができます。
* タスクの完了 / 未完了の切り替えができます。
* フィルター機能（すべて / 未完了 / 完了）があります。
* localStorageによるデータ永続化に対応しています（リロード後も保持されます）。
* 期限（締切日）の設定をすることができます。
* タスクの編集をすることができます

----

## 使用技術
HTML / CSS / JavaScript（Vanilla JS） / localStorage

----

## 工夫した点
* フィルター機能を実装し、状態に応じた表示切り替えができるようにし、ユーザーがタスクを管理しやすいようにしました。
* completedフラグでタスクの状態を管理できるようにしています。
* タスク追加後に現在のフィルター状態によっては表示されない問題がありました。そのため、追加時にフィルターを「すべて」に戻すことで、
　ユーザーが混乱しないようにしました。
* CSSで全て、完了、未完了のうち選択したものに色を付けることでアクティブ状態を視覚的に分かりやすく表現しました。
* タスクの期限をもたせることで、実用性を向上させました。
* 期限切れのタスクを赤、期限日のタスクをオレンジに色を付けることで
ユーザーがタスクを視覚的に管理できるようにしました。
